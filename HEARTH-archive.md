- **v333:** **Fix: per-person packing circles reverting to a single checkbox ~1s after toggling on (Cathal's bug report, flagged at v332; root-caused and fixed on Fable 17/07/2026).** NOT the accepted v326 two-partner race, and not `mergeListItems`'s fault — a **Firebase data-normalisation echo**: Firebase Realtime DB does not store empty objects, so pushing an item with `checks:{}` ("opted in, nothing ticked" — exactly what the item-sheet toggle writes for a not-yet-packed item) comes back off the device's OWN SSE echo with the `checks` field missing but the **same** `updated` stamp; the merge's remote-wins-on-tie rule then deletes the field and `renderList` drops back to the single checkbox. Timing fits: 800ms push debounce + PUT + echo ≈ the observed ~1s. This breaks the `handleHouseholdEvent` design assumption that "our own echo is an idempotent no-op" — the echo is only idempotent when the pushed JSON survives Firebase unchanged. Ticking a circle directly (`togglePersonCheck`) never hit this: it always leaves ≥1 key and Firebase stores `false` values fine — only the *empty* object is stripped, which is why ticked circles survived while the fresh toggle-on reverted. **Fix (representational, no sync-machinery changes):** "opted in, nothing ticked" is now stored as explicit `{c:false,p:false,b:false}` — same truthiness, same auto-pack logic, survives the round trip. New top-level `emptyPersonChecks()` helper (beside `clearPersonChecks`) used at all four `{}` writers: the item-sheet toggle-on (which also now normalises a pre-fix persisted `{}` via an `!Object.keys(...).length` guard), `clearPersonChecks` (covers the three revive paths), Reopen-all's inline `{checks:{}}`, and `togglePersonCheck`'s init. The latter three were the same latent bug — an opted-in item surviving "Unpack all" or a revive would have silently lost its circles on the next echo. `mergeListItems`, the tie-break, and the push/apply paths are deliberately untouched: a merge-side patch can't distinguish a stripped `{}` from a genuine toggle-off deletion, so it would break opt-out propagation. Mixed-version note: a partner device still on ≤v332 doing "Unpack all" can still strip circles until it updates — self-heals once both sides are on v333.
