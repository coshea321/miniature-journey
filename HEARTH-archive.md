# Hearth — Changelog archive (v230–v299)

Detailed changelog entries older than the newest ~10 versions live here, moved out of `HEARTH-notes.md` on 05/07/2026 to keep the per-session read small. **Read this file only when a specific old version is referenced** — day-to-day sessions don't need it. A condensed one-line index of every version stays in `HEARTH-notes.md` ("Shipped" section).

Maintenance: when a new version's detailed entry is added to `HEARTH-notes.md`, move the oldest detailed entry from there to the TOP of the list below, keeping this file newest-first.

- **v299:** **Sync-health panel in Settings (Tier 2 item 2; suggested by reviews 3 & 4).** New "📡 Sync health" button (under Cloud backups) opens a read-only diagnostics dialog: Status (signed in / logged out / offline), This device (online/offline via `navigator.onLine`), Household (paired / this-device-only), Live updates per channel (personal + household EventSource `readyState` → live / connecting / reconnecting / off), Last sent up, Last received, Pending deletions (total tombstone count across all `fl4_tomb_*`), Last cloud backup. A "Sync now" button force-fetches+pushes both channels and reopens the panel after 1.2s. **New observational state `_syncStats` {lastPersonalPush, lastHouseholdPush, lastEvent}** — set on the actual successful PUT (`.then(r=>r.ok)`) in `pushPersonal`/`pushHousehold` and at the top of both SSE handlers; nothing depends on it, it's display-only. New helpers: `syncAgo` (reuses `medAgoStr`), `countAllTombs`, `esState`, `openSyncHealthSheet`. Dialog at z-index 890 (same as cloud-backups sheet, below confirmDialog 900). No data-model/sync-behaviour change.
- **v298:** **Trips departure countdown badge (Tier 2 item 1 — the v275-deferred "countdown to departure").** New teal pill on each trip card: "Today" / "Tomorrow" / "In N days" for upcoming departures, "Now" while a trip is in progress (start passed, end still future), nothing for past trips. Two new top-level helpers: `tripDaysUntil(dateStr)` (whole-day delta from local-midnight today to a "YYYY-MM-DD" date, DST-safe because both endpoints are local midnight; null if unparseable) and `tripCountdownBadge(t)` (returns the pill HTML or ""). Wired into `tripCardHTML` as a flex sibling of the name/date block, so it shows in both Upcoming and Past sections but self-suppresses for past trips. No data-model change — reads existing `t.start`/`t.end`. Home dashboard "next departure" line NOT done here (that's part of the Tier 2 Today-card work).
- **v297:** **Train duplicate-function dedupe (external-review Tier 1, item 3 — closes the Tier 1 plan).** The seven remaining top-level Train helpers defined TWICE (`trainTodayStr`, `trainShiftDay`, `trainFmtDate`, `trainFmtDateLong`, `timerCircle`, `wireInputs`, `wireWorkoutLog` — remnants of the old "38 duplicates" incident) each had their **first** copy deleted. Method: every pair was extracted and diffed first — all seven byte-identical — and since both copies were top-level in the same script scope, function-declaration hoisting means the second always won at runtime, making the first copies provably dead. Zero behaviour change by construction; the trap it removes is "edited the first copy → silent no-op". `doSave` ×3 deliberately kept: those are nested closures in three different editors, not duplicates. ~78 lines deleted.
- **v296:** **Field-wise sync merges — Baby + Trips (external-review Tier 1, item 2).** Closes the "hand-typed field list silently wipes fields it doesn't name" class (the old babySex bug). (1) **applyPersonal baby merge:** known collections keep their existing union semantics (growth by date, medicine/milestones by id, teeth object-merge, bags remote-wins); every OTHER key now carries through generically — local wins when set (non-empty), remote fills gaps. (2) **applyHousehold baby:** was a **wholesale replace** (worse than the review thought — local unpushed baby changes could be dropped entirely); now `Object.assign({}, local, remote)` — remote still wins for every key it carries (household deletion semantics unchanged) but local-only keys survive. (3) **mergeTripsData:** trip-level merge now builds from both copies (winner's keys win, loser fills gaps) instead of a hand-typed `{id,name,start,end,updated}`; booking-level newest-wins now fills keys that are **undefined on the winner** from the loser — so an older-app-version device editing a booking can no longer wipe fields it predates (boarding/gate/seats being the live example). Deliberately `undefined`-only: a cleared field is `""` (defined) and stays cleared. **Known remaining limitation (accepted, needs baby tombstones to fix properly):** concurrent edits to the same baby COLLECTION across the household channel still resolve remote-wins (e.g. a dose logged locally in the window before a partner push could be dropped from the household copy, though the personal channel's union usually self-heals it). Documented, not fixed here.
- **v295:** **SW "lie-fi" fix + tombstones 90d (external-review Tier 1, item 1).** The v286 network-first shell had no timeout: on a connection that's present but not passing data, the app sat near-blank until the browser's own network timeout (up to ~60s) before cache fallback. Now the shell fetch races a **3.5s timer** — network wins if it answers in time (fresh-on-open behaviour unchanged), otherwise the cached shell is served and any late network response still lands in the cache for next open; with no cached copy at all (first ever visit) it keeps waiting on the network as before. Tombstone purge extended 30d → 90d (drawer-bound-tablet scenario; tombstones are tiny).

- **v291:** **Medicine dose-interval advisory + weight-date (council item 3 — design confirmed via 3 AskUserQuestion calls, all recommendations accepted).** SAFETY-SENSITIVE; advisory only, never a block. Reference intervals (shown to the user, never silently applied): **paracetamol ≥4h gap / max 4 in 24h; ibuprofen ≥6h / max 3 in 24h**; Antibiotic gets no rule. Three layers: (1) **always-visible "Last logged" summary** above the chips for each medicine present in the log — neutral grey normally, **amber** when inside the minimum gap or at the 24h max; shows ago-time, clock time, dose, and dose-count when ≥2 in 24h. (3) **One OK/Cancel confirm at log time** when the entered dose is inside the gap or over the max — checked against the **entered timestamp** (nearest same-medicine dose by absolute distance), so backfilling an old dose compares to its own neighbours, not to "now". Name matching is substring-based: calpol/paracetamol and nurofen/ibuprofen, so free-typed entries count. **Wording rule (from the council dissent, non-negotiable): every message says "logged"** ("There's a Calpol dose *logged* only 1h 40m from this one…", "Based only on logged doses — check the leaflet") so a message can never read as permission when a dose simply wasn't logged. Also: **dose calculator now shows the weight's date** — `latestBabyWeight()` replaced by `latestBabyWeightRec()` (returns the growth record); the working line reads "8.4kg (logged 12 Jun) x 15mg/kg ~ …" so a stale weight is self-evident. - **v288:** **Cloud backup restore + import overhaul.** New **"☁️ Cloud backups" button in Settings** (under the Last-cloud-backup line) opens a dialog listing the stored snapshots (shallow-keys GET, newest first, friendly date labels); tap **Restore** → `confirmDialog` → the snapshot is fetched and merged in, then both sync channels push and the current view re-renders. - **v287:** **Daily automatic cloud backup.** Once a day, ~8s after login (and on resume-from-background >30s), each device quietly PUTs the full export payload to the user's own private Firebase area (`/hearth/users/<user>/backups/<YYYY-MM-DD>`). Date-keyed → a second device the same day is an idempotent overwrite; keeps the newest **7** snapshots (`CLOUD_BACKUP_KEEP`), pruning older via a shallow-keys GET + DELETEs after each successful write. Both accounts back up independently (free redundancy). Silent on any failure — retries next open, logs "cloud backup skipped" to console only. Settings shows **"Last cloud backup: <date time>"** under the existing Last-backup line (`fl4_last_cloudbackup`, transient device-local, NOT exported). `exportAllData` was refactored: payload construction extracted to **`buildExportPayload()`** (top-level), shared by the file export and cloud backup so coverage can never drift — the coverage-checklist comment moved with it. **Manual export nudge relaxed 7 → 30 days** (design decision: the downloaded file now only guards against losing Firebase itself; cloud backup covers everyday risks). **Design confirmed with Cathal up front** (three AskUserQuestion calls): backup writer ships alone with the **restore UI as a follow-up version (v288 next)**; nudge stretched to 30 days (not kept at 7, not removed); 7 dailies retention (weekly tail rejected). **Restore path until v288:** ask Claude Code to pull a snapshot from Firebase, or use the Firebase console. **Not live-tested against Firebase from the build sandbox** — same REST/auth pattern as `pushPersonal` (`userUrl` + `?auth=` token, no-token fallback for legacy keyless setups); live app is the real confirmation.
- **v286:** **Offline app-shell fix (sw.js).** The fetch handler used to *deliberately not intercept* `/` and `/index.html` ("always fetch fresh") — so although both were cached at install, they were never served from cache, and an installed PWA opened with no network showed a browser error page instead of the app. Now the shell (navigation requests + `/` + `/index.html`) is **network-first with cache fallback**: online behaviour is identical to before (fresh fetch every open, so updates still land immediately), but on network failure the cached copy is served (with `ignoreSearch` + `./index.html` as the final fallback). Each successful shell fetch also refreshes the cached copy, so the offline fallback tracks the live version. Other same-origin assets stay cache-first; `sw.js` itself stays never-intercepted. Found by the v285 full-app council review (unanimous #1 priority). Accepted trade-off (flagged by Devil's Advocate): when *offline*, a newly-activated SW can serve a one-version-stale shell — harmless here since versions are backward-compatible over localStorage.

- **v264:** **Weight-based medicine dosing.** Medicine chips now compute a default dose from her **latest logged weight** (`latestBabyWeight()`, new top-level helper — newest growth record with a numeric `weight`).
